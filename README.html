Overview What Rele Is

OpenClaw is more capable than most hosted AI products — mobile nodes, browser automation, cron jobs, multi-agent routing, real channel integration, self-extending agents. But it requires Node 22+, a terminal, port forwarding, bot tokens, daemon management, and a mental model of what a "gateway" even is.

Rele is the way normal people access OpenClaw. Not a simplified version — the full thing, without the wall in front of it.

The pitch: "OpenClaw is incredible. You've never heard of it because you can't set it up. Rele fixes that."

The competition isn't ChatGPT. It's the Hetzner VPS they haven't gotten around to setting up yet.

Infrastructure Auth

Auth is handled by Neon Auth. The Next.js app handles auth server-side and client-side. The Hono API verifies JWTs from Neon Auth on every request using JWKS.

Methods enabled: Email / password. Google OAuth and GitHub OAuth coming soon.

Variable Service Notes
NEON_AUTH_BASE_URL web Neon Auth base URL · server-side
NEXT_PUBLIC_NEON_AUTH_URL web Neon Auth URL · client-side
NEON_AUTH_COOKIE_SECRET web Cookie signing secret
GATE_URL web Gate API URL · server-only · never prefix with NEXT_PUBLIC_
NEON_AUTH_URL gate Used by auth middleware for JWKS verification

Gate auth middleware (JWT verification via JWKS): 

// gate/src/middleware/auth.ts
const JWKS = jose.createRemoteJWKSet(
  new URL(`${process.env.NEON_AUTH_URL}/.well-known/jwks.json`)
);

const authMiddleware = async (c, next) => {
  const token = c.req.header("Authorization")?.split(" ")[1];
  const { payload } = await jose.jwtVerify(token, JWKS, {
    issuer: new URL(process.env.NEON_AUTH_URL).origin,
  });
  c.set("userId", payload.sub);
  await next();
};
Infrastructure Database

Postgres via Neon (serverless). One project covers all environments. The development instance lives on Neon — local dev points directly to it, no local Postgres needed.

Variable Service Notes
DATABASE_URL gate Neon connection string · used by gate to connect
Infrastructure Sandboxes — to build

Each user gets a Docker container on Fly.io running a pre-configured OpenClaw instance. Containers are provisioned on signup via the Fly.io Machines API and managed by gate.

This is the next major build milestone. The plan may evolve — Fly.io Machines API is the current approach.

Development Local Dev

No Docker Compose. Both services start from the monorepo root. The Neon dev database and Neon Auth are used directly during local development — no local instances needed.

Prerequisites: 

bun
access to github.com/relegate-to/configuration (private)

Setup: 

git clone git@github.com:relegate-to/rele.git
cd rele
bash dev.sh setup    # applies env files from configuration repo

Run: 

bun dev    # starts web (:3000) and gate (:3001) concurrently
Service URL Notes
web localhost:3000 Next.js frontend
gate localhost:3001 Hono API

Env management via dev.sh:

Command Description
bash dev.sh setup Apply env files on a fresh clone
bash dev.sh env:pull Pull and apply latest env files
bash dev.sh env:push Push local env changes to configuration repo

Each service has its own .env.local at its directory root. These are gitignored and never committed.
